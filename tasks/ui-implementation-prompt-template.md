# UI Implementation from PRD — Meta Prompt Template

## Usage
Copy the prompt below, customize `[PLACEHOLDERS]`, paste into Claude Code in the **target project**.

The PRD file must already exist (generated by `ui-exploration-prompt-template.md`).

---

### Output Organization Rules

- **One PRD per feature/plugin** — All PRD content for a feature must be in a SINGLE markdown file, not spread across multiple files.
- **Feature-scoped directories** — Each feature gets its own directory under the project root. Example: if working on "Funnel", all output goes under `funnel/` directory.
  - `funnel/PRD.md` — Single combined PRD
  - Do NOT create multiple PRD files (PRD-styles.md, PRD-interactions.md, etc.) — merge everything into one.
  - Do NOT create `prd-assets/` directories or save screenshots — CSS values and HTML structure are sufficient.
- **No root-level PRD sprawl** — Never save PRDs, assets, or temporary files in the project root directory. Always use a feature-scoped subdirectory.
- **Naming convention**: `{feature-name}/PRD.md` (e.g., `cohort-drawer/PRD.md`, `funnel/PRD.md`)

---

## The Prompt

```
Implement a pixel-perfect reimplementation of [FEATURE_NAME] using Tailwind CSS, based entirely on the attached PRD document. The result must be visually indistinguishable from the original [APP_NAME] UI.

## Inputs
- PRD file: [PATH_TO_PRD e.g. ./cohort-drawer/PRD.md]
- Output directory: [OUTPUT_DIR]

## The Golden Rule

The PRD is the ONLY source of truth. Every color, every pixel, every spacing value, every interaction, every state — it's all documented in the PRD. You do NOT invent, assume, or "improve" anything. If the PRD says `padding: 12px`, you write `padding: 12px`. If the PRD documents a CSS value, your output must match that value exactly. Period.

---

## Team Structure

Operate as a virtual engineering team. Use subagents aggressively — maximize parallelism at every step.

### Tech Lead (Orchestrator — YOU)
- Reads the entire PRD, understands all 6 parts
- Decomposes the UI into parallel work units based on PRD Part 4 (DOM Structure)
- Assigns each work unit to a Countly UI Developer subagent
- Reviews all outputs against PRD Part 2 (Styles) and documented CSS values
- Integrates pieces into the final working UI
- Iterates until pixel-perfect

### Countly UI Developer (Subagents — multiple, running in parallel)
- Each developer owns ONE section of the UI
- Reads the PRD sections relevant to their assigned section
- Studies the documented CSS values and DOM structure for their section carefully
- Implements structure, styles, behavior, interactions, and localization — all from the PRD
- Does NOT invent anything not documented in the PRD

---

## How to Work

### Step 1: Read the Entire PRD

Read the PRD file completely. It has 7 parts — understand each one:

| PRD Part | What It Tells You | How You Use It |
|----------|-------------------|----------------|
| Part 1 — Functional Behavior | What the UI does: dropdowns, fields, conditions, validation | Implement all logic exactly as documented |
| Part 2 — Design System & Styles | Every CSS value: colors, typography, spacing, shadows, transitions | Translate each value to Tailwind classes |
| Part 3 — Interaction Flows | What happens on click/type/select: transitions, state changes, cascading dependencies | Wire all interactions exactly as documented |
| Part 4 — HTML Structure & Classes | DOM skeleton: tags, nesting, class names, data-test-ids | Your component structure mirrors this exactly |
| Part 5 — Cee vs Original Comparison | Known visual differences between AI chat form and original | Checklist of issues to avoid |
| Part 6 — Localization & Data Rules | Which strings are `[STATIC/i18n]`, `[DYNAMIC/DB]`, `[STATIC/FIXED]` | Handle each category correctly |

### Step 2: Study the CSS Values and DOM Structure

Review ALL documented CSS values in PRD Part 2 and DOM structures in Part 4. These are your implementation targets.

For each UI element:
- What CSS values are documented?
- What state variants exist (default, hover, focus, active, disabled, error)?
- What DOM nesting and class names are specified?
- What borders, shadows, transitions are defined?

PRD Part 2 gives you the exact computed values; Part 4 gives you the exact structure.

### Step 3: Set Up Design Tokens

From PRD Part 2, extract ALL design tokens and add them to your Tailwind configuration:

- **Colors**: Every hex value with its usage context
- **Typography**: Every font-size / font-weight / line-height combination
- **Spacing**: Every padding / margin value
- **Border radius**: Every unique value
- **Box shadows**: Every shadow definition
- **Transitions**: Duration, easing, properties
- **Z-index**: Every layer

Use project-prefixed names (e.g. `cly-green`, `cly-title`) for values that repeat across the UI.

### Step 4: Decompose & Implement in Parallel

Look at PRD Part 4 (DOM Structure). Each major section of the UI is a **work unit**.

**Launch a subagent for EVERY section simultaneously.** If the PRD documents 8 sections, run 8 subagents in parallel. If it documents 5, run 5.

Each subagent receives the relevant slices from ALL PRD parts:

```
You are a Countly UI Developer. Implement the [SECTION_NAME] section.

Your PRD inputs:
- DOM structure (Part 4): [paste the relevant section]
- CSS styles (Part 2): [paste the relevant styles]
- Behavior (Part 1): [paste the relevant logic]
- Interactions (Part 3): [paste the relevant flows]
- Localization (Part 6): [paste the relevant tags]
Rules:
1. Match the DOM hierarchy from Part 4 exactly — same nesting, same order
2. Translate EVERY CSS value from Part 2 to Tailwind — no guessing
3. Implement ALL state variants: default, hover, focus, active, disabled, error
4. Implement conditional rendering exactly as Part 1 documents
5. Match transitions/animations from Part 3
6. Preserve every data-test-id from Part 4
7. Handle localization per Part 6 tags
8. Match every documented CSS value exactly — your output must be pixel-perfect
9. Do NOT invent behavior or styles not in the PRD
```

### Step 5: Integrate

After all subagents complete:
1. Assemble sections into the full UI
2. Wire shared state across sections
3. Verify the complete UI renders correctly
4. Compare the assembled result against the PRD documented values — again

### Step 6: Review Against PRD Values

Go through every element documented in PRD Part 2 (Styles) one by one:
1. Read the documented CSS values for each element
2. Look at your implementation for the same element
3. If any value differs — fix it immediately
4. Repeat until every documented value matches

This is a loop. Keep going until there are zero differences.

## Parallel Execution Strategy

| Task | Parallel? | Why |
|------|-----------|-----|
| Read PRD | NO | Must read before anything |
| Design token setup | NO | Foundation for all styling |
| **Section implementation** | **YES — ALL SECTIONS** | Each section is independent |
| Integration / wiring | NO | Needs all sections complete |
| PRD value review | NO | Needs assembled UI |
| Fix iterations | NO | Sequential by nature |

**Maximum parallelism happens at Step 4** — every UI section gets its own subagent, all running at the same time. This is where 80% of the work happens. Do NOT implement sections sequentially.

---

## Execution Rules

1. **PRD is truth** — every value comes from the PRD, never from your imagination
2. **CSS values are targets** — match every documented value exactly, do not deviate
3. **Maximize parallelism** — every independent section runs as its own subagent, simultaneously
4. **Tailwind only** — no original UI library classes, pure Tailwind utilities + custom config tokens
5. **All states** — default, hover, focus, active, disabled, error — all must match PRD Part 2
6. **Preserve data-test-ids** — every test-id from PRD Part 4 must exist in output
7. **Fix until perfect** — iterate on PRD values until zero differences remain
8. **No over-engineering** — don't add features, don't refactor, don't "improve" — just match the PRD
9. **Respect localization tags** — `[STATIC/i18n]` gets localized, `[DYNAMIC/DB]` stays raw, `[STATIC/FIXED]` stays hardcoded
```
